AutoheightSyncState#

Ƭ AutoheightSyncState: "init" | "syncing" | "synced"

The state of synchronization between viewport and content size:

  • init: the initial, "onMount" state;
  • syncing: the content size is being determined;
  • synced: the viewport size has been adjusted to content size.

HTMLDimensionsImplementation#

Ƭ HTMLDimensionsImplementation: "resize" | "mutation" | "polling"

The script will check for different APIs in order to implement the notification of HTML dimensions changes. By order of preference: ResizeObserver (resize), MutationObserver (mutation) and polling.

Variables#

ForceElementSizeFeature#

• Const ForceElementSizeFeature: FeatureConstructor<ForceElementSizeOptions>

This feature sets element size programmatically and only once, when DOMContentLoaded has been fired.

ForceResponsiveViewportFeature#

• Const ForceResponsiveViewportFeature: FeatureConstructor<ForceResponsiveViewportOptions>

This feature inserts or replace a meta element looking like:

<meta name="viewport" content="width=device-width" />

This will guarantee that the layout viewport will match device-width (hence, “responsive”). Minimum scale is locked to 1, but you can customize maximum scale to allow pinch-zoom gestures. See MDN—Using the viewport meta tag ...

HandleElementCSSBoxFeature#

• Const HandleElementCSSBoxFeature: FeatureConstructor<HandleElementCSSBoxDimensionsOptions, { onDOMElementCSSBoxDimensions: PropDefinition<"onDOMElementCSSBoxDimensions", (e: ElementCSSBoxDimensions) => void> }>

This feature enables receiving the CSS Box dimensions of an element in the WebView pixels unit. The first element matching the provided tagName is retained. A new event will be triggered on every resize.

remarks If you are looking for the document content size, use HandleHTMLDimensionsFeature instead.

HandleHTMLDimensionsFeature#

• Const HandleHTMLDimensionsFeature: FeatureConstructor<HandleHTMLDimensionsOptions, { onDOMHTMLDimensions: PropDefinition<"onDOMHTMLDimensions", (d: HTMLDimensions) => void> }>

This feature enables receiving various dimensions relative to the layout. The events will only be fired when a change is observed to either the layout or the content size.

remarks If you need to guarantee that 1 CSS pixel = 1 Device pixel, you should use this feature with a meta tag viewport setting width to device width. See ForceResponsiveViewportFeature

HandleHashChangeFeature#

• Const HandleHashChangeFeature: FeatureConstructor<HandleHashChangeOptions, { onDOMHashChange: PropDefinition<"onDOMHashChange", (e: HashChangeEvent) => void> }>

This feature allows to intercept hashchange events, when the hash fragment of the URL changes. This could happen when the user clicks on anchors like this:

<a href="#subresource">Let's Jump to Subresource</a>

Or when JavaScript code imperatively changes the hash fragment of current location. See MDN—hashchange event.

HandleLinkPressFeature#

• Const HandleLinkPressFeature: FeatureConstructor<LinkPressOptions, { onDOMLinkPress: PropDefinition<"onDOMLinkPress", (t: LinkPressTarget) => void> }>

This feature allows to intercept clicks on anchors such as:

<a href="https://domain.com/">Let's Travel</a>

remarks By default, it will prevent the click from propagating. But you can disable this option.

If you need to intercept click on hash fragments, consider HandleHashChangeFeature instead.

HandleVisualViewportFeature#

• Const HandleVisualViewportFeature: FeatureConstructor<{}, { onDOMVisualViewport: PropDefinition<"onDOMVisualViewport", (d: VisualViewportDimensions) => void> }>

This feature adds a handler triggered when the visual viewport changes. It requires VisualViewport API support on browsers (iOS Safari 13 and Android WebView 62). See VisualViewport API.

useAutoheight#

▸ useAutoheight<S>(params: AutoheightParams<S>): AutoheightState<S>

Requires HandleHTMLDimensionsFeature and recommends ForceResponsiveViewportFeature.

remarks This hook has caveats you must understand:

  • Because the viewport height is now bound to the content heigh, you cannot and must not have an element which height depends on viewport, such as when using vh unit or height: 100%; on body. That will either create an infinite loop, or a zero-height page (this happens for Wikipedia). Hence, it is strongly advised that you use autoheight only with content you have been able to test. This can be worked around by forcing body height to 'auto', see ForceElementSizeFeature.
  • When the user clicks to fragment links within the same page (e.g, “#help”), there will be no scrolling, because this is handled by WebView on overflow, and there is no such overflow when in autoheight mode.

beta

Type parameters:#

NameType
SWebshellProps<MinimalWebViewProps, [FeatureInstanceOf<typeof HandleHTMLDimensionsFeature>]>

Parameters:#

NameTypeDescription
paramsAutoheightParams<S>The parameters to specify autoheight behavior.

Returns: AutoheightState<S>

  • An object to implement autoheight behavior.
